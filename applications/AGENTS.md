# Applications

## Purpose

The deployable NukeIDE shells: the Theia browser application, the Electron desktop application, and the all-in-one Docker deployment.

## Ownership

All files under `applications/` except generated artifacts (`*/lib/`, `*/node_modules/`, `*/src-gen/`, `gen-webpack*.js`, `*.tsbuildinfo`, `electron/dist/`, `docker/workspace/`).

## Local Contracts

- `browser/` and `electron/` are Theia application packages. Their `package.json` dependency lists decide which extensions ship — registering a new extension means adding it to **both**.
- `src-gen/`, `gen-webpack.config.js`, `gen-webpack.node.config.js`, and `lib/` are produced by the Theia generator/webpack during builds. Never edit them; regenerate instead.
- `browser/` scripts: `bundle` (rebuild + `theia build`), `start`, `watch`. Driven from the repo root via `yarn build:browser` / `yarn start:browser` / `yarn watch:browser`.
- `electron/` scripts: `bundle` / `bundle:prod` (rebuild → preload generation → `theia build` → `patch-electron-main`), `start`, `dist` (electron-builder). Driven from the root via `yarn build:electron(:prod)`, `yarn start:electron`, `yarn dist`, `yarn package:electron(:all)`.
- Electron `rebuild` skips `cpu-features`: it is an optional native dependency of `ssh2` that frequently fails to build in CI (missing generated `buildcheck.gypi`), and `ssh2` falls back to a pure-JS implementation without it. `rebuild` also does not build `@theia/ffmpeg`'s `ffmpeg.node` (not in Theia's default module list); that addon comes from the implicit node-gyp install during `yarn install`, and `theia build`'s `checkFfmpeg` hard-crashes without it — CI rebuilds it explicitly after cache restores.
- Electron packaging on CI (`.github/workflows/build.yml`) explicitly runs `node node_modules/electron/install.js` after `yarn install`; the shared `node_modules` cache may have been created by a script-skipping install and therefore lack the `electron/dist` binary required by `theia build`.
- `electron/src-gen/backend/electron-main.js` is patched after every bundle by `scripts/patch-electron-main.js` (sets `THEIA_DEFAULT_PLUGINS` so bundled VS Code extensions load in packaged apps); the bundle scripts already run it.
- `resources/preload.html` (repo root `resources/`) is generated by `scripts/generate-preload.js` from `resources/preload.template.html` with version + tips injected; edit the template or `resources/tips.yml`, never the output.
- `docker/` builds the all-in-one container: stage 1 (node:22-bookworm) installs `libx11-dev`/`libxkbfile-dev` (required to compile native node modules such as `native-keymap`) and compiles the browser app; stage 2 (miniforge) installs X/GL system libraries (`libgl1`, `libxft2`, … — gmsh and OCP/cadquery link them even headless) and provides the `ide` conda env defined in `environment.yml` (Python 3.13, Node 22, OpenMC, Trame/ParaView/VTK, gmsh/CadQuery/pythonocc-core, DAGMC toolchain, headless VTK). In `environment.yml`: `openmc` and `moab` (provides `pymoab`) come from the `https://shimwell.github.io/wheels` extra pip index — the same index nuke-core health checks use; the openmc wheel bundles libopenmc and the DAGMC-enabled `openmc` executable, so conda-forge openmc is intentionally not used (a pip-only environment can then replicate the image for tests); `pydagmc` (pip, from GitHub) must stay pinned to a commit; and `paraview`/`vtk` (conda) plus `cadquery`/`cadquery-ocp`/`cadquery_vtk` (pip) must stay pinned to the verified combo — pip's `cadquery_vtk` shares the `vtkmodules` namespace with conda's `vtk` and clobbers conda files during env creation, which is why the Dockerfile force-reinstalls the conda VTK family (`paraview`, `vtk`, `vtk-base`, `vtk-io-ffmpeg`) after `conda env create`; that ordering step must not be removed. Base images use fully-qualified registry names (`docker.io/...`) so builds work on podman hosts without short-name aliasing. Stage 1 also prunes dev-only npm packages (`electron`, `electron-builder`, `lerna`, `typescript`, `prettier`) and download caches — that cleanup must stay in stage 1 so the bytes never reach the final image layers. `compose.yml` mounts `docker/workspace/` as `/root/workspace` and a named volume (`nuke-ide-config`) as `/root/.nuke-ide` for settings persistence; real OpenMC runs need a cross-section library mounted with `OPENMC_CROSS_SECTIONS` set (commented example in `compose.yml`). The image is built and smoke-tested by `.github/workflows/docker.yml`.

## Work Guidance

- Adding or removing an extension: update both app `package.json` files, then re-bundle. Extension code changes alone only need the extension's `build` plus an app re-bundle.
- Electron packaging runs on CI for Linux/Windows/macOS (`.github/workflows/build.yml`); local packaging is `yarn package:electron`.
- The Docker image targets the browser app only — do not wire Electron into it.
- Runtime Python dependencies for extension backends belong in `docker/environment.yml` (and `docs/installation.md`), not in the root `requirements-dev.txt`, which is dev/test-only.

## Verification

```bash
yarn build:browser      # webpack bundle of the browser app
yarn build:electron     # webpack bundle of the Electron app
yarn docker:build       # container image
```
